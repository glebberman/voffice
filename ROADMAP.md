# voffice — план разработки

Поэтапный план развития от текущего MVP (движение + presence + proximity-чат)
до полноценного Gather-подобного виртуального офиса. Этапы упорядочены так,
чтобы каждый давал ощутимый результат, а дорогие вещи опирались на уже готовое.

Текущая архитектура-опора: presence-канал `office`, client-to-client whispers
(pos/move/chat), Pixi-сцена, карта-JSON, Docker Compose (app/reverb/vite/queue).

---

## Этап 1 — Социальные фичи поверх whisper'ов ⏱ ~2–3 вечера

Быстрые победы: всё делается по образцу существующих whisper-событий,
без изменений на сервере.

- [ ] **1.1 Эмодзи-реакции** — хоткеи 1–5 и панель: 👋 ❤️ 😂 🎉 👍.
      Whisper `react`, всплывающая анимация над аватаром (Pixi), автоскрытие.
- [ ] **1.2 Статусы** — available / busy / не беспокоить + авто-away по
      неактивности (visibilitychange + idle-таймер). Точка-индикатор на
      аватаре и в списке «Онлайн». Whisper `status` + поле в heartbeat `pos`.
- [ ] **1.3 Приватные зоны чата** — в переговорке слышно только тех, кто
      внутри неё (правило «одна зона» вместо радиуса); в опенспейсе — радиус,
      как сейчас. Бейдж «Вы в зоне: Переговорка» уже есть — расширить логику
      фильтра в `use-office.ts`.

**Критерий готовности:** два клиента обмениваются реакциями и статусами;
сообщение из переговорки не слышно снаружи и наоборот.

---

## Этап 2 — Персистентность и первые серверные события ⏱ ~2 вечера

Здесь впервые задействуем сервер и queue-контейнер по назначению.

- [ ] **2.1 Чат комнаты с историей** — таблица `messages`
      (сразу с `room_id`, default 1 — задел под этап 3), `POST /api/messages`,
      серверный broadcast `MessageSent` через Reverb, история последних 50
      при входе. Вкладки чата: «Рядом» (эфемерный, как сейчас) | «Комната»
      (персистентный).
- [ ] **2.2 Сохранение позиции** — throttled-запись последней позиции игрока
      (при уходе со страницы + раз в ~30 сек), спавн на прежнем месте.

**Критерий готовности:** перезагрузка возвращает игрока на место и
показывает историю чата комнаты.

---

## Этап 3 — Мир: объекты, комнаты, порталы ⏱ ~3–4 вечера

- [ ] **3.1 Интерактивные объекты** — слой объектов в карте: `{тип, url, тайл}`.
      Подошёл к доске → подсказка «нажмите X» → модалка с iframe
      (Miro/Excalidraw, YouTube, Google Docs). Подсветка объекта рядом.
- [ ] **3.2 Комнаты и порталы** — модели `Room`/`Map` (карта переезжает из
      hardcode в JSON-поле), канал `presence-room.{id}`, плитки-порталы
      с переходом между комнатами. Лобби со списком комнат.
- [ ] **3.3 (опц.) Фоновые звуки зон** — эмбиент в лаунже, затихающий
      с расстоянием.

**Критерий готовности:** из офиса через портал можно уйти в другую комнату —
presence и чат переключаются; у стола открывается общая доска.

## Этап 4 — Аватары и навигация ⏱ ~2–3 вечера

- [ ] **4.1 Кастомизация аватара** — спрайтшит (цвет/скин/причёска или
      готовые LPC-спрайты вместо кружков), выбор в настройках профиля,
      поле в `users`, анимация ходьбы.
- [ ] **4.2 Follow / Locate** — клик по коллеге в списке → стрелка-указатель
      к нему; режим «следовать».
- [ ] **4.3 Ring/buzz** — «позвать» человека: whisper + браузерная нотификация.

---

## Этап 5 — Видео/аудио по близости (WebRTC mesh) ⏱ ~4–6 вечеров

Самый большой шаг. Mesh-топология: до ~5–6 участников одновременно, чего
для пет-проекта достаточно.

- [ ] **5.1 Сигналинг поверх whisper'ов** — offer/answer/ICE-кандидаты
      через события `rtc:*` (инфраструктура уже есть).
- [ ] **5.2 Автосоединение по близости** — вход в радиус/зону → peer
      connection, выход → разрыв. Громкость затухает с расстоянием.
- [ ] **5.3 UI звонка** — плитки видео поверх сцены, mute/камера off,
      индикатор «кто говорит».
- [ ] **5.4 Screen sharing** — getDisplayMedia в существующий peer.
- [ ] **5.5 coturn в compose** — TURN-сервер для работы вне localhost.

**Критерий готовности:** трое в переговорке видят и слышат друг друга;
уход из зоны рвёт соединение; демка экрана видна остальным.

**Риск/развилка:** если понадобится >6 человек в звонке — миграция на SFU
(LiveKit контейнером в compose + токен-эндпоинт в Laravel); сигналинг 5.1
при этом выбрасывается, UI 5.3 остаётся.

## Сквозные принципы

- Эфемерное (позиции, реакции, typing) — только whispers, минуя сервер.
  Персистентное (сообщения, карты, профили) — обычный Laravel: HTTP + БД +
  серверный broadcast.
- Каждый этап заканчивается рабочей демкой на двух клиентах и зелёным CI.
- SQLite менять на Postgres только когда появится реальная причина
  (полнотекстовый поиск по чату, конкурентная запись) — добавить сервис
  в compose на тот момент несложно.
