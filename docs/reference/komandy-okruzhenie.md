# Справочник: команды и переменные окружения

## Запуск

| Команда                                 | Что делает                                                             |
| --------------------------------------- | ---------------------------------------------------------------------- |
| `docker compose up -d`                  | весь стек в контейнерах: app (8000), reverb (8080), vite (5173)        |
| `docker compose --profile webrtc up -d` | то же + coturn (TURN/STUN для WebRTC вне LAN)                          |
| `docker compose down`                   | остановить                                                             |
| `composer run dev`                      | запуск без Docker: serve + pail + vite + reverb одной командой         |

## Разработка

| Команда                                  | Что делает                         |
| ---------------------------------------- | ---------------------------------- |
| `npm run dev`                            | Vite в режиме разработки (HMR)     |
| `npm run build`                          | продакшн-сборка фронтенда          |
| `npm run format`                         | Prettier исправляет форматирование |
| `npm run format:check`                   | проверка форматирования (как в CI) |
| `npm run lint`                           | ESLint без правок (как в CI)       |
| `npm run lint:fix`                       | ESLint с автоисправлением          |
| `npm run types`                          | проверка типов (`tsc --noEmit`)    |
| `php artisan migrate`                    | миграции                           |
| `php artisan db:seed`                    | сидеры (предметы, комнаты, юзеры)  |
| `php artisan db:seed --class=RoomSeeder` | только комнаты                     |

В Docker выполняйте artisan-команды внутри контейнера:
`docker compose exec app php artisan …`.

## Правки из браузера и файлы в репозитории

Карты и каталог предметов правятся в браузере и живут в БД, а
`resources/maps/*.json` и `resources/props.json` — стартовое наполнение сидеров.
Правка из редактора **не попадает в файл сама** и исчезнет при пересоздании базы.

Замыкает круг команда:

```bash
php artisan voffice:export        # выгрузить БД → resources/
npm run format                    # привести JSON к стилю репозитория
git diff resources/               # посмотреть, что изменилось, и закоммитить
```

`--check` ничего не пишет, только показывает разошедшиеся файлы и возвращает
ненулевой код — удобно спросить «а всё ли выгружено?» перед коммитом. Сравнение
идёт по данным, а не по тексту: файлы отформатированы Prettier, а PHP печатает
JSON иначе, и посимвольное сравнение ругалось бы на каждый файл.

`--to=<каталог>` выгружает копию в сторону, не трогая репозиторий.

## Качество фронтенда

Линтер — **ESLint** с типизированными правилами (`strictTypeChecked` +
`stylisticTypeChecked` из typescript-eslint). Они видят типы, а не только
синтаксис: ловят забытые `await`, небезопасные `any` из библиотек и условия,
которые всегда истинны.

Правила держатся без единого `eslint-disable`. Два правила настроены, а не
отключены: `no-confusing-void-expression` с `ignoreArrowShorthand` (иначе
каждый `onClick={() => setX(...)}` требует скобок) и
`restrict-template-expressions` с `allowNumber`. Выключено одно —
`consistent-type-definitions`: оно требует `interface`, а `useForm` от Inertia
принимает только `type` с индексной сигнатурой.

`tsconfig.json` покрывает и тесты с конфигами, так что `npm run types`
проверяет их тоже.

В CI все три команды проверяют, а не правят: `format:check`, `lint` без
`--fix`, `types`. Раньше `format` и `lint` запускались в режиме записи —
файлы молча переформатировались, результат выбрасывался, и сломать сборку
стилем было невозможно.

## Качество PHP-кода

| Команда         | Что делает                                            |
| --------------- | ----------------------------------------------------- |
| `composer lint` | проверка стиля и статический анализ — то же, что в CI |
| `composer fix`  | Pint исправляет стиль на месте                        |
| `composer stan` | только PHPStan, с прогрессом                          |

Стиль держит **Pint** (это php-cs-fixer с ларавеловским пресетом) по правилам из
`pint.json`. В CI он запускается с `--test`: сборка падает на неотформатированном
коде, а не переформатирует его молча.

Статический анализ — **PHPStan + Larastan**, конфиг в `phpstan.neon`.
**Максимальный уровень 10**, без baseline и без `ignoreErrors`.

Что понадобилось, чтобы его взять:

- `App\Support\CurrentUser::of($request)` вместо `$request->user()->…` — за
  middleware `auth` пользователь есть всегда, и 401 на его отсутствие честнее
  ветки, которую никогда не выполнят;
- `App\Support\Wardrobe` и `PropType::catalogue()` разбирают JSON в типы один
  раз, с внятной ошибкой при кривом файле, — вместо `mixed` по всему коду;
- аксессоры `Message::authorName()` и `sentAt()`: `timestamps()` заводит колонки
  nullable, поэтому у модели `created_at` — `?Carbon`;
- значения запроса читаются типизированно (`$request->integer('x')`,
  `$request->string('id')->toString()`), а не из результата `validate()` —
  он отдаёт массив неизвестных значений. Правила при этом никуда не делись:
  `validate()` по-прежнему вызывается и по-прежнему бракует запрос;
- у колбэков каналов и уведомлений проставлены типы параметров.

Держать 10-й уровень стоит внимания при правках: любое обращение к невалидному
`mixed` теперь ошибка сборки, а не сюрприз в рантайме.

Анализатору нужен `--memory-limit=1G`: в дефолтные 128 МБ он не укладывается.

## Тесты

| Команда                   | Что проверяет                                            |
| ------------------------- | -------------------------------------------------------- |
| `./vendor/bin/phpunit`    | PHP: маршруты, авторизация каналов, API, редактор карт   |
| `npm test` (`vitest run`) | JS: карты, коллизии, `canHear`, гардероб, BFS, proximity |

CI (`.github/workflows/`) гоняет оба набора на каждый push в `main`.

## Ключевые переменные окружения

### Приложение и Docker

| Переменная             | Назначение                                   |
| ---------------------- | -------------------------------------------- |
| `APP_PORT`             | порт приложения на хосте (по умолчанию 8000) |
| `WWWUSER`, `WWWGROUP`  | uid/gid для прав на volume (1000)            |
| `BROADCAST_CONNECTION` | драйвер вещания (`reverb`)                   |
| `QUEUE_CONNECTION`     | драйвер очереди (`database`); сейчас не используется — фоновых задач нет |

### Reverb (WebSocket)

| Переменная                                    | Назначение                             |
| --------------------------------------------- | -------------------------------------- |
| `REVERB_APP_ID/KEY/SECRET`                    | ключи приложения Reverb                |
| `REVERB_HOST`, `REVERB_PORT`, `REVERB_SCHEME` | адрес для серверной публикации         |
| `VITE_REVERB_APP_KEY/HOST/PORT/SCHEME`        | адрес для браузера (вшивается в бандл) |

### WebRTC (ICE)

| Переменная                          | Назначение                                  |
| ----------------------------------- | ------------------------------------------- |
| `VITE_STUN_URL`                     | STUN-сервер (пусто = только host-кандидаты) |
| `VITE_TURN_URL/USERNAME/CREDENTIAL` | TURN-сервер для работы за NAT               |
| `TURN_USERNAME`, `TURN_CREDENTIAL`  | учётные данные coturn                       |

> `VITE_*` вшиваются на этапе сборки — после их изменения нужен перезапуск Vite
> (dev) или пересборка (`npm run build`).
