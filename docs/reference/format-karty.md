# Справочник: формат карты и тайлы

Карта комнаты — JSON-объект. Исходники лежат в `resources/maps/*.json` (имя файла =
`slug` комнаты), сидер заливает их в поле `rooms.map`. На клиенте карту разбирает
`makeMap(data)` из [`resources/js/game/map.ts`](../../resources/js/game/map.ts).

## Структура

```json
{
    "rows": ["#####", "#...#", "#####"],
    "spawn": { "x": 2, "y": 1 },
    "zones": [{ "name": "Кухня", "x1": 1, "y1": 1, "x2": 3, "y2": 1, "isPrivate": false, "kind": "kitchen" }],
    "objects": [{ "id": "board", "type": "board", "label": "Доска", "url": "https://…", "x": 2, "y": 1 }],
    "portals": [{ "x": 1, "y": 1, "to": "coworking", "label": "В коворкинг", "tx": 3, "ty": 3 }],
    "props": [{ "id": "cabinet-1", "type": "cabinet", "x": 6, "y": 6 }]
}
```

| Поле      | Тип           | Описание                                                          |
| --------- | ------------- | ----------------------------------------------------------------- |
| `rows`    | `string[]`    | сетка тайлов; **все строки одинаковой ширины**                    |
| `spawn`   | `{x,y}`       | стартовая клетка; **должна быть проходимой**                      |
| `zones`   | `Zone[]`      | прямоугольные зоны (`x1,y1,x2,y2`, `name`, `isPrivate?`, `kind?`) |
| `objects` | `MapObject[]` | интерактивные объекты                                             |
| `portals` | `Portal[]`    | телепорты в другие комнаты                                        |
| `props`   | `Prop[]`      | многоклеточные предметы обстановки (необязательное поле)          |
| `doors`   | `Door[]`      | двери в проёмах (необязательное поле)                             |

Размер тайла — 32×32 px (`TILE`). Клетка `(x, y)` = символ `rows[y][x]`.

## Символы тайлов

| Символ | Тайл              | Проходимый |
| :----: | ----------------- | :--------: |
|  `.`   | пол               |     да     |
|  `#`   | стена             |    нет     |
|  `:`   | плитка кухни      |     да     |
|  `,`   | ковёр переговорки |     да     |
|  `;`   | ковёр лаунжа      |     да     |
|  `*`   | spotlight (сцена) |     да     |
|  `D`   | рабочий стол      |    нет     |
|  `K`   | кухонная стойка   |    нет     |
|  `T`   | стол переговорки  |    нет     |
|  `S`   | диван             |    нет     |
|  `P`   | растение          |    нет     |

Проходимые символы: `.` `:` `,` `;` `*`. Полный список для палитры редактора —
`TILE_CHARS` в `map.ts`.

## Зоны

Зона — прямоугольник `[x1,y1]..[x2,y2]` включительно. Если `isPrivate: true`
(например, переговорка), чат внутри зоны слышен только тем, кто в той же зоне, и
не выходит наружу — независимо от радиуса. Обычная зона нужна только для подписи.

Необязательный `kind` (`meeting`, `kitchen`, `lounge`, `openspace`, `stage`) —
тип помещения: редактор карт красит зону цветом пресета, слоги совпадают с осью
«тип помещения» каталога предметов. Игра `kind` пока игнорирует; отсутствие =
«своя». Зоны рисуются и правятся в редакторе (инструмент «Область»); сервер
проверяет, что прямоугольник в границах карты и углы не перевёрнуты.

## Spotlight `*`

Проходимый тайл. Игрок, стоящий на нём, вещает в чат на **всю комнату**, минуя
правило близости. Метод `map.isSpotlight(x, y)`.

## Объекты

| Поле   | Значения                              |
| ------ | ------------------------------------- |
| `type` | `board` \| `video` \| `map` \| `link` |
| `url`  | адрес встроенного окна (iframe)       |
| `x, y` | клетка объекта                        |

Подойдя ближе `OBJECT_RADIUS` (1.6 тайла) и нажав `X`, игрок открывает `url` в модалке.

## Порталы

| Поле     | Описание                                       |
| -------- | ---------------------------------------------- |
| `x, y`   | клетка портала в текущей комнате               |
| `to`     | slug целевой комнаты (**должна существовать**) |
| `tx, ty` | клетка прибытия в целевой комнате              |
| `label`  | подпись                                        |

## Предметы (`props`)

Тайлы `D K T S P` занимают ровно одну клетку. Предметы крупнее (стол, шкаф,
телевизор) описываются отдельным списком `props` — в карте хранится только тип и
позиция, а размеры берутся из каталога — таблицы `prop_types`
(см. [базу данных](baza-dannyh.md#prop_types)). Каталог правится в браузере на
`/props` ([как завести предмет](../how-to/zavesti-predmet.md)); стартовое
наполнение — [`resources/props.json`](../../resources/props.json), из которого
таблицу заполняет `PropTypeSeeder`.

Клиент не хранит каталог у себя: он приезжает с сервера вместе с картой
(`propTypes` в props страницы), а `makeMap(data, catalogue)` принимает его
явным параметром — иначе «зашитая» копия разъехалась бы с тем, по чему
валидирует карту сервер.

| Поле       | Описание                                                       |
| ---------- | -------------------------------------------------------------- |
| `id`       | произвольный уникальный идентификатор                          |
| `type`     | ключ из `props.json` (`cabinet`, `desk`, `tv`, …)              |
| `x, y`     | **левая верхняя клетка основания**                             |
| `dir`      | сторона, которой стоит предмет; нет поля — значит `south`      |
| `settings` | настройки поведения инстанса (embed → `{label, url}`) — опц.   |

**Поведение и настройки.** У типа может быть `behavior` (реестр в коде,
`game/behaviors.ts`): первое — `embed`, встраиваемый URL. Стоя в зоне
взаимодействия такого предмета, игрок нажимает `X` и открывает `settings.url` в
модалке (обобщение старых `objects`). Настройки хранятся у предмета в карте
(`settings`), потому что у каждой поставленной доски свой адрес; форму проверяет
сервер (`PropBehaviors`). Нет `settings` = предмет не настроен и неинтерактивен.

Запись каталога — предмет с набором **ориентаций** (сторон, которыми он может
стоять): у каждой свой регион на листе и своя геометрия, потому что при повороте
меняется и footprint (стол 4×1 становится 1×2):

```json
"cabinet": {
    "label": "Шкаф (высокий)",
    "orientations": {
        "south": { "sheet": "office/Desk, Ornate.png", "sx": 96, "sy": 0, "w": 2, "h": 1, "tall": 2 }
    }
}
```

| Поле     | Описание                                                      |
| -------- | ------------------------------------------------------------- |
| `dir`    | ключ ориентации: `south` (дефолт), `west`, `east`, `north`    |
| `sheet`  | путь внутри `public/assets/lpc/`                              |
| `sx, sy` | левый верхний угол спрайта в тайлсете (px)                    |
| `w, h`   | размер **основания** в тайлах — эти клетки непроходимы        |
| `tall`   | сколько тайлов спрайта поднимается **над** основанием         |
| `states` | регионы состояний: `{"on": {sheet, sx, sy}, …}` — опционально |
| `interaction` | зона взаимодействия: `[{dx, dy}, …]` от origin — опционально |

**Состояния** — именованные регионы поверх ориентации (телевизор включён /
выключен / без сигнала): у региона свой лист и угол, а размер всегда
`w×(h+tall)` своей ориентации, потому что состояние меняет картинку, но не
геометрию. Имена состояний общие для всех сторон типа. `defaultState` на
предмете — что рисуется, пока предметом не пользуются; отсутствие поля =
состояний нет. Живое переключение состояний приедет вместе с `prop_states`.

**Зона взаимодействия** (`interaction`) — клетки, стоя на которых персонаж
сможет пользоваться предметом: список смещений `{dx, dy}` относительно origin
(левого верхнего тайла основания), отрицательные допустимы. Зона своя на каждую
сторону — поворот телевизора разворачивает и её. На клетках основания зоны быть
не может (на них не стоят). Отсутствие/пустой список = предмет не интерактивен;
само взаимодействие в игре подключится с поведениями.

Если у предмета на карте указана сторона, которой в каталоге уже нет, клиент и
сервер одинаково откатываются: запрошенная сторона → `south` → первая
существующая (`propOrientation` в `game/props.ts` и
`MapUpdateRequest::orientationOf`).

### Высокие предметы и проход за ними

Спрайт режется на две части. Нижние `h` тайлов (основание) рисуются **под**
игроками и блокируют проход. Верхние `tall` тайлов рисуются **над** игроками, но
проход не блокируют: персонаж проходит за предметом и частично скрывается им.

Чтобы игрок не терялся за высоким спрайтом, вокруг **своего** персонажа
вырезается овал прозрачности: верхний слой маскируется по инверсии, а копия того
же слоя показывается внутри овала с `alpha ≈ 0.32`.

Овал включается, только когда персонаж **стоит на клетке, накрытой верхним
слоем** (`map.isOverhead(x, y)` — часть предмета «в воздухе» или верхушка
стены), и режет **только тот предмет, чья это клетка**: маска своя у каждого
куска overhead. Иначе дырка появлялась бы везде, где овал геометрически задел
спрайт, — например в предмете, мимо которого просто проходят сбоку.

Сам овал вдобавок срезан сверху по линии макушки (`root.y - 48`, ровно граница
тайловой строки). Это не украшение: просвечивать нужно только то, что стоит на
уровне персонажа или ниже и потому его перекрывает. Всё, что стоит выше по карте,
находится «за спиной», и дырка в нём показывала бы изнанку сцены. Проверять
каждый объект не требуется — предмет, стоящий выше персонажа, целиком лежит выше
линии макушки. Подробнее — [docs/explanation/resheniya.md](../explanation/resheniya.md).

### Горизонтальные стены в два тайла

Стена `#` занимает одну клетку логически, но рисуется в два тайла: над каждой
стеной дорисовывается «верхушка» (`map.isWallCrown(x, y)` — клетка проходима, а
под ней `#`). Верхушка попадает в тот же верхний слой, что и высокие предметы:
за ней можно ходить, и её так же прорезает овал.

## Двери (`doors`)

Дверь стоит на **проходимой** клетке — обычно в проёме стены. Закрытая не
пропускает; запертую нельзя открыть, пока её не отопрут со стороны замка.

| Поле   | Описание                                                           |
| ------ | ------------------------------------------------------------------ |
| `id`   | произвольный уникальный идентификатор                              |
| `x, y` | клетка двери                                                       |
| `lock` | `north` \| `south` \| `west` \| `east` \| `null` — где висит замок |

```json
{ "id": "kitchen-door", "x": 13, "y": 5, "lock": "north" }
```

`lock` — сторона, с которой дверь запирают и отпирают: игрок должен стоять на
соседней клетке именно с этой стороны. Замок работает как ключ с одной стороны —
запертую дверь не открыть ни снаружи, ни изнутри, так что запереться внутри
можно.

### Состояние двери — не в карте

Открыта дверь или заперта, хранится в таблице `door_states`
(см. [базу данных](baza-dannyh.md)), а не в `rooms.map`: карту правит
администратор в редакторе, а двери дёргают игроки, и сохранение карты затирало
бы чужие двери. Двери без строки в таблице считаются открытыми.

Изменение уходит всем в комнате событием `door.changed`
(см. [реальное время](realtime.md)).

### Что видно за закрытой дверью

Клиент считает обходом в ширину, куда персонаж может дойти, **вместе с
границей** этой области: своя мебель, свои стены и закрытая дверь тоже видны,
хотя пройти по ним нельзя. Без границы в тень уходил бы каждый непроходимый
тайл, включая столы в той самой комнате, где персонаж стоит.

Всё за пределами набора затемняется: пол и мебель уходят в тень, чужие
персонажи там не рисуются вместе с именами и пузырями, а чат поблизости сквозь
закрытую дверь не проходит.

От этого же набора зависит и отрисовка стен: верхушка, попавшая в доступное
помещение, не рисуется, чтобы стена ближе к камере не загораживала комнату.

## Ограничения размера

Максимум — **512×512 тайлов** (262 тыс. клеток). Предел выбран как «максимум для
адекватной работы»: карта такого размера весит ~258 КБ сырого JSON (~15–30 КБ в
gzip, тайловые строки жмутся отлично), а рендер и редактор не зависят от размера
карты, потому что оба показывают только видимую область.

Проверяется в `MapUpdateRequest` (`map.rows` ≤ 512 элементов, каждая строка
≤ 512 символов) и в `MAX_MAP_SIZE` из `map.ts`.

## Инварианты (проверяются и в PHP, и в JS)

- все строки `rows` одинаковой ширины;
- периметр карты — сплошная стена (`#`);
- `spawn` на проходимой клетке;
- объекты и порталы в границах карты;
- `portals.*.to` — существующая комната;
- `props.*.type` — существующий `slug` в `prop_types`;
- основание предмета целиком внутри карты, а высокой части хватает места сверху
  (`y - tall >= 0`);
- дверь стоит на проходимой клетке, и на одной клетке не больше одной двери.

Серверная проверка — `app/Http/Requests/MapUpdateRequest.php`; клиентская —
`tests/js/map.test.ts` и `tests/js/props.test.ts`.
