# Справочник: Realtime — каналы и события

Realtime построен на Laravel Reverb (WebSocket) и Laravel Echo. Есть два вида
сообщений: **whisper** (клиент→клиент, минуя сервер) для эфемерного и серверный
**broadcast** для персистентного.

## Каналы

| Канал                  | Тип      | Авторизация (`routes/channels.php`)                                                     |
| ---------------------- | -------- | --------------------------------------------------------------------------------------- |
| `room.{roomId}`        | presence | любой авторизованный; отдаёт `{ id, name, avatar }`; для несуществующей комнаты — отказ |
| `App.Models.User.{id}` | private  | стандартный канал пользователя (из стартер-кита)                                        |

Клиент присоединяется к комнате как `echo.join('room.'+roomId)`. Данные
присутствия (`{ id, name, avatar }`) приходят в колбэках `here/joining/leaving`.

## Whisper-события (клиент → клиент)

Отправляются и принимаются в `resources/js/hooks/use-office.ts` на канале
`room.{id}`. Сервер их не видит и не хранит.

| Событие  | Нагрузка                        | Смысл                                   |
| -------- | ------------------------------- | --------------------------------------- |
| `pos`    | `{ id, x, y, dir, st?, call? }` | периодический heartbeat позиции/статуса |
| `move`   | `{ id, x, y, dir, st?, call? }` | шаг игрока                              |
| `chat`   | `{ id, name, text, x, y }`      | сообщение proximity-чата                |
| `react`  | `{ id, emoji }`                 | эмодзи-реакция                          |
| `status` | `{ id, status }`                | смена статуса (available/busy/dnd/away) |
| `look`   | `{ id, avatar }`                | смена внешности (живое применение)      |
| `buzz`   | `{ from, name, to }`            | «позвать» участника                     |
| `call`   | `{ id, inCall }`                | вход/выход из звонка                    |
| `rtc`    | `{ from, to, signal }`          | WebRTC-сигналинг (offer/answer/ICE)     |

Правила приёма: события со своим `id` от других вкладок игнорируются там, где
это важно (позиция, статус); `rtc`/`buzz` адресные — проверяется поле `to`;
`react`/`status` валидируются по белым спискам.

## Серверные broadcast-события

| Событие                  | Канал                  | Alias           | Через очередь          |
| ------------------------ | ---------------------- | --------------- | ---------------------- |
| `App\Events\MessageSent` | `room.{id}` (presence) | `.message.sent` | да (`ShouldBroadcast`) |

`MessageSent` порождается в `POST /messages`, проходит через контейнер `queue`,
доставляется всем в комнате, кроме сокета отправителя (`->toOthers()` +
`X-Socket-ID`). Нагрузка: `{ id, userId, name, body, at }`.

## Почему так разделено

- **whisper** — мгновенно, дёшево, без нагрузки на сервер и БД, но эфемерно
  (офлайн не увидит).
- **broadcast через очередь** — переживает перезагрузку, доступно офлайн через
  историю, но дороже.

Ориентир: позиции/реакции/статусы/сигналинг — whisper; чат комнаты — broadcast.
Подробнее — [Пояснение про архитектуру](../explanation/arhitektura.md).

## Конфигурация Reverb

Ключи и адреса — в `.env` (`REVERB_*`, `VITE_REVERB_*`). В Docker внутри сети
серверная публикация идёт на хост `reverb`, а браузер подключается к
`localhost:8080`. Клиентские события включены (`accept_client_events_from=members`).
